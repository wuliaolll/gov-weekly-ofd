## 二、技术栈与依赖

| 模块 | 技术 | 说明 |
|------|------|------|
| Web 框架 | Flask 3.1.x | 页面路由与 API |
| HTML 解析 | BeautifulSoup4 + lxml | 栏目页与详情页结构解析 |
| 抓取方式 | `curl.exe` 子进程 | 规避 Python SSL 兼容问题 |
| DOCX 生成 | python-docx | 公文排版输出 Word |
| PDF 排版 | ReportLab | OFD 前置中间版式层 |
| OFD 转换 | easyofd + 自定义补丁 | 由 `pdf2ofd.py` 注入增强能力 |
| 中文分词 | jieba 0.42.1 | 标题智能折行（词性标注+语义断点） |
| 定时任务 | APScheduler | 每日定时采集 |
| 前端交互 | Tailwind CDN + Alpine.js | 轻量页面与实时状态刷新 |

说明：`pdf2ofd.py` 依赖 PyMuPDF(`fitz`) 与 Pillow，用于高保真提取字符、路径和图像资源并写入 OFD 结构。

## 三、核心流程设计

### 3.1 采集与生成流程

1. 前端调用 `POST /api/collect` 启动后台线程。
2. 后端解析栏目列表，或按 `report_url` 处理指定周报。
3. 解析周报详情，按日期分组并抽取领导活动。
4. 对每条活动抓取详情正文，提取段落与对齐信息。
5. 生成 DOCX 与 OFD，并落盘至统一目录结构。
6. 前端轮询 `GET /api/status` 展示进度与日志。

### 3.2 OFD 生成链路

当前实现链路：

`正文段落 -> ReportLab PDF(矢量文本) -> PDF2OFDConverter(pdf2ofd.py) -> OFD`

关键点：

1. `doc_generator.py` 负责公文版式、字体映射与中西文混排处理。
2. 标题与正文均采用中英文混排：中文用对应字体（标题方正小标宋简体、正文仿宋GB2312），数字和英文字母统一用 Times New Roman。
3. `pdf2ofd.py` 对 easyofd 进行 monkey patch：
   - 提升文字定位精度（基于字符级坐标与 DeltaX）。
   - 统一字体名称映射与资源组织。
   - 修正路径/图形/图像转换细节。
   - 生成标准 OFD ZIP 结构。

### 3.3 标题智能折行

标题超过 18 个 CJK 字符宽度时，通过 `_clean_title_text()` + `_smart_title_lines()` 两步处理：

#### 3.3.1 标题文本清理（`_clean_title_text()`）

在折行前先对从网页提取的标题文本做如下清理：

1. **去除换行变体**：将 `\r\n`、`\r`、`\n`、`\u2028`（行分隔符）、`\u2029`（段分隔符）、`\u0085`（NEL）统一替换为空字符串（CJK 汉字直接连接，不需要分隔符）。
2. **Zs 类 Unicode 空格归一化**：将所有 Unicode 空格类字符（如 `\u00A0` NBSP、`\u2002` en-space、`\u2003` em-space、`\u3000` 全角空格等）统一替换为普通 ASCII 空格，避免在中文字体下渲染为缺字形圆圈。
3. **圆圈样占位符清除**：移除 `°`（U+00B0）、`˚`（U+02DA）、`∘`（U+2218）、`◦`（U+25E6）、`○`（U+25CB）、`◌`（U+25CC）等网页复制噪声字符。
4. **伪换行符号清除**：移除 `↵`（U+21B5）、`⏎`（U+23CE）、`␊`（U+240A）、`␍`（U+240D）等可视化换行符号。
5. **零宽/方向控制字符清除**：移除软连字符、零宽字符、BOM、BiDi 嵌入控制符等。
6. **最终兜底**：通过 `unicodedata.category()` 过滤所有剩余 Cc/Cf 类控制字符（保留普通空格），合并连续空格并去除首尾空格。

> `scraper.py` 的 `fetch_article_content()` 亦在标题构建后立即做一次简化版清理（Zs 归一化 + Cc/Cf 过滤），确保来自 HTML 结构噪声的异常字符在进入生成器前即被清除。

#### 3.3.2 语义折行（`_smart_title_lines()`）

**前置：末尾人名+动词检测**

在一切折行逻辑前，用 jieba POS 扫描标题末尾是否为"人名序列 + 出席/参加/列席/主持/致辞/陪同"形式。若命中，将尾部强制剥离为独立最后一行，对主体部分递归折行。扫描采用两轮策略：
- 严格轮：只接受 jieba 标注为 `nr`/`nz` 的人名 token
- 宽松轮（兜底）：接受 2-4 字 CJK 且非明确停止词性（`v/a/d/p/c` 等）的 token，但须含至少一个 `nr`/`nz` token 才采信

**空格分隔标题快速路径**

若标题含空格且**所有片段均不超过 18 字符宽**，按空格拆为短语片段后，通过递归枚举合并方案取最少行、行宽最均匀的方案。若任意片段超宽，则落入 jieba DP。

**jieba 语义 DP 路径**（片段超宽或无空格标题统一走此路径）：

- **jieba 分词 + 词性标注**：注册政务复合词（科技创新、高质量发展、营商环境、支点建设、决定性进展等）避免误拆。
- **短语合并（Phase 1）**：
  1. 修饰词(≤2字)+动词(≤2字) → 复合动词（牢固+树立→牢固树立）
  2. "X性"结尾词(≤4字)+名词 → 整体（决定性+进展→决定性进展）
  3. 内容词(≤2字)+修饰词(≤2字) → 并列修饰语（科学+务实→科学务实）
- **断点语义评分（Phase 2）**：
  - 从句边界（动宾完成+新动词短语开始）给予 -15 奖励
  - 粘连词性（副词/介词/连词等）+12 惩罚
  - 动宾结构不拆 +5 惩罚
  - **中文排版禁则**：断点后为行首禁则字符（右引号/右括号/句末标点）或断点前为行尾禁则字符（左引号/左括号）时，惩罚 +50
- **DP 搜索**：以语义断点分数为主、行宽均匀性为辅，搜索最优分行方案。

#### 3.3.3 DOCX 标题段落结构

`generate_docx()` 中，`_smart_title_lines()` 返回的每一行都生成一个独立 `<w:p>` 段落，而非使用 `<w:br/>` 软换行：

- 每行：居中对齐、固定 32pt 行距、`space_before=0`、`space_after=0`。
- Word 中每行显示为正常段落结束符（¶），而非软换行符（↵），保证导出和打印的段落结构符合公文规范。

## 四、解析策略设计

### 4.1 栏目页解析

1. 入口 URL 默认：`https://www.hubei.gov.cn/hbfb/zwzb/index.shtml`。
2. 选择器：`ul.hbgov-newslist-itemheight-18px li`。
3. 输出字段：`title`、`url`、`pub_date`。

### 4.2 周报页活动抽取

1. 日期分隔：匹配 `X月X日`。
2. 活动分隔：匹配 `^►+`。
3. 领导识别：
   - 独立领导名段落。
   - 箭头后紧跟领导名。
   - 领导名开头的活动标题行。
4. 链接识别：`详情<<` 优先，同时兼容 `.shtml/.html`。

### 4.3 正文页清洗

1. 容器优先级：`.bt_content`、`#myText`、`.TRS_Editor`、`.article-content`、`.hbgov-article-content`。
2. 过滤规则：编辑/责编/审核/来源、短署名、图解行。
3. 保留段落对齐（居中/左对齐）给文档生成器。

## 五、目录与文件设计

### 5.1 输出目录

```
output/门户网站周报/
└── {年份}年/
    └── {周报日期范围}/
        └── {领导姓名}{职务}/
            └── {活动日期}/
                ├── {活动标题}.docx
                └── {活动标题}.ofd
```

### 5.2 命名与安全

1. 文件/目录名移除非法字符：`\\ / : * ? " < > |`。
2. 文件名最大 80 字符。
3. 下载接口使用路径归属校验，禁止目录穿越。

### 5.3 打包下载规范

1. 按“年份/周期”目录打包。
2. zip 内结构与生成目录一致。
3. 不再单独分 `docx/` 与 `ofd/` 顶层目录。

## 七、前端页面设计

### 7.1 视觉与布局

1. 红色公文风配色：`#C41E24` 主色、`#B8860B` 辅色。
2. 页面模块：
   - 配置区（URL、定时、开关、保存、预览、全量采集）。
   - 栏目预览区（按期触发）。
   - 任务状态区（进度条+日志）。
   - 文件树区（年份/周期/领导/日期展开，单文件与打包下载）。

### 7.2 交互

1. 任务启动后轮询状态（约 1.5s）。
2. 任务结束自动刷新文件树。
3. 状态日志自动滚动到底部。

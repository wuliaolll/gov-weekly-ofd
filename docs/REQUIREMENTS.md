# 政务周报 OFD 生成器 — 需求文档（现状版）

## 一、项目背景

湖北省政府门户网站周报需要按公文规范批量生成电子归档文件。项目目标是在不引入重型系统的前提下，实现“抓取-解析-排版-产出-下载”闭环，支持 DOCX 与 OFD 双格式交付。

本文档基于当前代码实现更新，作为当前版本的需求基线。

## 二、当前目标与范围

### 2.1 当前版本目标

1. 提供可用的 Web 操作界面，支持配置、预览、采集与下载。
2. 从栏目页自动发现周报链接并解析详情页活动。
3. 自动抓取活动详情正文并生成 DOCX 与 OFD。
4. 按固定目录结构落盘，并支持单文件下载与按周期打包下载。
5. 支持每日定时任务与手动触发任务。

### 2.2 当前版本不在范围内

1. 多用户权限体系。
2. 数据库存储与任务历史持久化。
3. 分布式任务队列与失败重试编排。
4. 复杂模板管理后台。

## 三、功能需求（按实现状态）

### 3.1 已实现功能

1. 配置管理
   - 配置项：栏目 URL、定时小时/分钟、是否启用自动采集。
   - 接口：`GET /api/config`、`POST /api/config`。

2. 栏目预览与任务触发
   - 支持预览栏目页解析结果。
   - 支持“全量采集生成”与“采集某一期”。
   - 接口：`POST /api/column-preview`、`POST /api/collect`。

3. 任务状态可视化
   - 返回运行状态、当前步骤、进度和日志。
   - 接口：`GET /api/status`。

4. 文件树浏览与下载
   - 以“年份-周期-领导-日期-文件”树形展示。
   - 支持单文件下载。
   - 接口：`GET /api/reports`、`GET /api/download/<path:filepath>`。

5. 周期打包下载
   - 支持按周期下载 zip。
   - 要求：zip 内部目录必须与生成目录一致，不单独区分 `docx/` 与 `ofd/` 顶层目录。
   - 接口：`GET /api/download-zip/<path:period_path>`。

6. 文稿生成
   - DOCX：`python-docx` 生成公文版式；标题每折行一行对应一个独立段落（`<w:p>`），而非软换行符（`<w:br/>`）。
   - OFD：ReportLab 生成 PDF 中间产物，再通过 `pdf2ofd.py` 的高保真转换器输出 OFD。
   - 标题清理：生成前通过 `_clean_title_text()` 去除 HTML 结构噪声（`\n`/`\r` 等换行变体、`\u00A0` 等 Zs 类 Unicode 空格、`○` 等圆圈占位符、零宽/BiDi 控制字符），避免中文字体下缺字形渲染为圆圈。

7. 定时任务
   - APScheduler 每日定时触发采集任务。

### 3.2 解析规则需求

1. 栏目页解析
   - 选择器：`ul.hbgov-newslist-itemheight-18px li`。
   - 提取字段：标题、链接、发布时间。

2. 周报详情页解析
   - 日期分隔：`X月X日`。
   - 活动分隔：支持 `►`/`►►`/`►►►`。
   - 领导识别：已知领导名单匹配，支持“独立领导名行”与“领导名开头标题行”。
   - 详情链接识别：优先 `详情<<`，兼容 `.shtml/.html` 文章链接。

3. 正文提取
   - 容器优先级：`.bt_content`、`#myText`、`.TRS_Editor`、`.article-content`、`.hbgov-article-content`。
   - 过滤：编辑/责编/审核/来源、短署名行、图解说明行。
   - 保留段落与居中信息用于排版。

## 四、文件组织要求

### 4.1 生成目录结构

```
output/门户网站周报/
└── {年份}年/
    └── {周报日期范围}/
        └── {领导姓名}{职务}/
            └── {活动日期}/
                ├── {活动标题}.docx
                └── {活动标题}.ofd
```

### 4.2 目录与文件命名规则

1. 年份由发布时间优先推断，缺失时从标题推断，仍缺失则回退当前年份。
2. 文件名与目录名需清洗非法字符：`\ / : * ? " < > |`。
3. 文件名最长 80 字符，避免路径过长。

### 4.3 打包下载规则

1. 打包单位：一个“年份/周期”目录。
2. zip 内目录与该周期下实际生成目录保持一致。
3. 同一活动目录下保留 `.docx` 与 `.ofd` 并存，不做格式分仓。

## 五、版面规范需求

### 5.1 页面设置

| 参数 | 值 |
|------|-----|
| 纸张大小 | A4 (210mm × 297mm) |
| 上边距 | 3.7cm |
| 下边距 | 3.0cm |
| 左边距 | 3.2cm |
| 右边距 | 3.2cm |

### 5.2 字体与字号

| 元素 | 字体 | 字号 |
|------|------|------|
| 标题（中文） | 方正小标宋简体 | 二号 (22pt) |
| 标题（数字/英文） | Times New Roman | 二号 (22pt) |
| 正文（中文） | 仿宋GB2312 | 小二号 (18pt) |
| 正文（数字/英文） | Times New Roman | 小二号 (18pt) |

说明：标题和正文中的数字、英文字母均通过 `_split_cn_en()` 拆分后单独设置 Times New Roman 字体，DOCX 和 OFD 两种格式保持一致。

### 5.3 段落要求

| 参数 | 值 |
|------|-----|
| 对齐 | 两端对齐（正文）/ 居中（特定段落） |
| 首行缩进 | 2字符 |
| 行距 | 固定值 32 磅 |

### 5.4 标题折行规则

1. 标题单行最多 18 个 CJK 字符宽度，超出时自动智能折行。
2. 折行前先做文本清理：去除所有换行变体（`\n`/`\r`/`\u2028` 等，替换为空字符串）、将 Zs 类 Unicode 空格归一为普通空格、去除圆圈占位符与零宽/控制字符。
3. 含空格的标题，且**所有按空格拆分的短语片段均不超过 18 字符宽**时：按空格拆分后递归枚举合并方案，取最少行+最均匀宽度方案，行内保留原始空格。若任一片段超宽则直接进入 jieba DP 路径。
4. 无空格标题或片段超宽时：基于 jieba 分词与语义断点 DP 搜索，保证短语完整性（如"科学务实""牢固树立""深入推进"不拆分）。

### 5.5 其他要求

1. 文本颜色黑色。
2. 标题后空一行。
3. 正文不加粗、不插图、不设置页码。

## 七、验收标准

1. 可通过页面完成配置保存、栏目预览、任务触发与文件下载。
2. 采集任务运行时可在状态面板看到实时日志和进度。
3. 生成目录符合“四层目录+双文件”规范。
4. 周期 zip 解压后目录与原生成目录一致。
5. DOCX 与 OFD 均可正常打开，文本可读，版式满足公文要求。
